# Structurizr DSL Output

`nfr-review arch --format dsl` generates a [Structurizr DSL](https://docs.structurizr.com/dsl) workspace file from the architecture scan. The DSL file is a human-readable, version-controllable representation of your system's architecture using the [C4 model](https://c4model.com/).

## Quick start

```bash
# Generate DSL alongside the default reports
nfr-review arch --format dsl /path/to/your/repo

# Generate only DSL output
nfr-review arch --format dsl --output-dir reports /path/to/your/repo

# Combine with other formats
nfr-review arch --format json --format md --format dsl /path/to/your/repo
```

The output file is written to `<output-dir>/arch-workspace.dsl`.

## Worked example: Spring Petclinic

Running against the [Spring Petclinic](https://github.com/spring-projects/spring-petclinic) repository:

```bash
nfr-review arch --format dsl ./spring-petclinic
```

Produces a workspace like this (abbreviated):

```dsl
workspace "Architecture -- spring-petclinic" "Auto-generated by nfr-review" {
    !identifiers hierarchical

    model {
        spring_petclinic = softwareSystem "spring-petclinic" "Primary system under analysis" {
            compose_spring_petclinic_mysql_5a694a = component "mysql" "Docker Compose service" "mysql"
            compose_spring_petclinic_postgres_b15904 = component "postgres" "Docker Compose service" "postgres"
            comp_spring_petclinic_998494 = component "spring-petclinic" "Root project component" "Java, Spring Boot"
        }
        infra_postgresql_a478b4 = softwareSystem "Postgresql" "Infrastructure: Postgresql" {
            tags "External"
        }
        infra_mysql_430005 = softwareSystem "Mysql" "Infrastructure: Mysql" {
            tags "External"
        }

        comp_spring_petclinic_998494 -> compose_spring_petclinic_mysql_5a694a "JDBC connection to mysql" "jdbc:mysql"
        comp_spring_petclinic_998494 -> infra_postgresql_a478b4 "JDBC connection to postgresql" "jdbc:postgresql"
    }

    views {
        systemLandscape "landscape" {
            include *
            autoLayout tb
        }
        container spring_petclinic "containers" {
            include *
            autoLayout lr
        }
        styles {
            element "External" {
                shape RoundedBox
                background #999999
                color #ffffff
            }
        }
    }
}
```

### What nfr-review detected

From the repository's source code and configuration, the scan extracted:

- **Components**: the Spring Boot application, plus MySQL and PostgreSQL Docker Compose services
- **External systems**: database infrastructure (MySQL, PostgreSQL, H2) identified from Maven dependencies and JDBC connection strings
- **Relationships**: JDBC connections inferred from `application.properties` and Docker Compose service links

## Worked example: Kubernetes microservices

For a Kubernetes-based project like [microservices-demo](https://github.com/GoogleCloudPlatform/microservices-demo):

```bash
nfr-review arch --format dsl ./microservices-demo
```

The scan detects 14 Kubernetes Deployment workloads as components, with relationships derived from:

- **Environment variables** referencing other services (e.g. `CART_SERVICE_ADDR`)
- **gRPC proto analysis** identifying service-to-service call graphs
- **Redis connections** from Kubernetes manifests

```dsl
model {
    microservices_demo = softwareSystem "microservices-demo" "Primary system under analysis" {
        k8s_microservices_demo_frontend_d31c8b = component "frontend" "Kubernetes Deployment workload" "frontend"
        k8s_microservices_demo_cartservice_07d8cc = component "cartservice" "Kubernetes Deployment workload" "cartservice"
        k8s_microservices_demo_checkoutservice_1f4fb5 = component "checkoutservice" "Kubernetes Deployment workload" "checkoutservice"
        k8s_microservices_demo_redis_cart_fa1f49 = component "redis-cart" "Kubernetes Deployment workload" "redis"
        // ... 10 more services
    }
    k8s_microservices_demo_frontend_d31c8b -> k8s_microservices_demo_cartservice_07d8cc "gRPC call from 'frontend' to 'CartService'" "grpc"
    k8s_microservices_demo_cartservice_07d8cc -> k8s_microservices_demo_redis_cart_fa1f49 "K8s Deployment 'cartservice' references 'redis-cart' via env REDIS_ADDR" "redis"
    // ... more relationships
}
```

## What the DSL contains

### Model elements

| C4 Level | Source | Example |
|----------|--------|---------|
| Software System | Primary repo + detected external infrastructure | `infra_postgresql = softwareSystem "Postgresql"` |
| Container | Internal services, databases, queues | K8s Deployments, Docker Compose services |
| Component | Code-level modules, libraries | Root project component with detected tech stack |

### Relationships

Relationships are extracted from:

- Maven/Gradle/pip/npm dependency analysis
- Kubernetes manifest env vars and service references
- Docker Compose service links and networks
- gRPC proto `import` and `service` definitions
- JDBC/database connection strings
- Integration points from static analysis

Each relationship includes a description and technology string (e.g. `"grpc"`, `"jdbc:mysql"`, `"redis"`).

### Views

The generated workspace always includes:

- **System Landscape** (`autoLayout tb`) -- top-down overview of all systems
- **Container view** (`autoLayout lr`) -- left-to-right view of containers within the primary system

### Tags and styles

| Tag | Applied to | Style |
|-----|-----------|-------|
| `External` | Infrastructure and cross-repo systems | Grey rounded box |
| `CrossRepo` | Relationships spanning repositories | Red dashed line |
| `dev` | Dev-only infrastructure (e.g. H2 Database) | (no special style) |

## Rendering architecture diagrams

The generated `.dsl` file is a complete Structurizr workspace. Several tools can render it into interactive diagrams or exportable images.

### Structurizr Lite (recommended for local use)

[Structurizr Lite](https://docs.structurizr.com/lite) is a free, local web app that renders DSL workspaces as interactive C4 diagrams.

**1. Start the container**, mounting the directory that contains your `arch-workspace.dsl`:

```bash
docker run -it --rm -p 8080:8080 \
  -v $(pwd)/reports:/usr/local/structurizr \
  structurizr/lite
```

**2. Open http://localhost:8080** in your browser. You'll see the generated views:

- **System Landscape** — all software systems and their relationships
- **Container view** — internal services, databases, and queues within the primary system

**3. Navigate between views** using the diagram selector in the top-left. Each view corresponds to a `views {}` block in the DSL.

**4. Edit the DSL** to refine your model. Structurizr Lite watches the file for changes — save the `.dsl` file and the browser auto-reloads. Common refinements:

```dsl
views {
    # Add a component view for a specific container
    component spring_petclinic.comp_spring_petclinic_998494 "components" {
        include *
        autoLayout lr
    }

    # Customise styles
    styles {
        element "External" {
            shape RoundedBox
            background #999999
            color #ffffff
        }
        element "Software System" {
            background #1168bd
            color #ffffff
        }
    }
}
```

**5. Export diagrams** from the browser UI — click any diagram, then use the PNG/SVG export button in the top-right toolbar.

### Structurizr CLI (batch export)

The [Structurizr CLI](https://docs.structurizr.com/cli) exports diagrams to multiple formats without a browser. Install it via Homebrew or download from GitHub:

```bash
brew install structurizr-cli
# or: download from https://github.com/structurizr/cli/releases
```

**Export to PlantUML** (renders via PlantUML server or local jar):

```bash
structurizr-cli export -workspace reports/arch-workspace.dsl -format plantuml -output diagrams/
# Produces: diagrams/structurizr-landscape.puml, diagrams/structurizr-containers.puml
```

**Export to Mermaid** (embeddable in GitHub markdown):

```bash
structurizr-cli export -workspace reports/arch-workspace.dsl -format mermaid -output diagrams/
```

**Export to PNG/SVG** via PlantUML:

```bash
# Export to PlantUML first, then render
structurizr-cli export -workspace reports/arch-workspace.dsl -format plantuml -output diagrams/
plantuml -tsvg diagrams/*.puml
```

**Export to D2** (for use with the [D2](https://d2lang.com/) renderer):

```bash
structurizr-cli export -workspace reports/arch-workspace.dsl -format d2 -output diagrams/
d2 diagrams/structurizr-landscape.d2 diagrams/landscape.svg
```

### CI integration

Generate and render architecture diagrams as part of your CI pipeline:

```yaml
# .github/workflows/architecture.yml
name: Architecture diagrams
on:
  push:
    branches: [main]

jobs:
  render:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - name: Install nfr-review
        run: pip install nfr-review

      - name: Generate DSL
        run: nfr-review arch --format dsl --output-dir reports .

      - name: Render diagrams
        run: |
          npm install -g @structurizr/cli
          structurizr-cli export -workspace reports/arch-workspace.dsl \
            -format plantuml -output diagrams/
          plantuml -tsvg diagrams/*.puml

      - name: Upload diagrams
        uses: actions/upload-artifact@v7
        with:
          name: architecture-diagrams
          path: diagrams/*.svg
```

### Structurizr Cloud / On-Premises

For team-shared interactive diagrams, push the workspace to [Structurizr Cloud](https://structurizr.com/) or a self-hosted [On-Premises](https://docs.structurizr.com/onpremises) instance:

```bash
structurizr-cli push -workspace reports/arch-workspace.dsl \
  -id <workspace-id> -key <api-key> -secret <api-secret>
```

This enables shared diagram browsing, version history, and diagram embedding via iframe.

### Version control

The DSL file is plain text and diffs cleanly in Git. Commit it alongside your source code to track architecture evolution over time:

```bash
git add reports/arch-workspace.dsl
git commit -m "docs: update architecture model"
```

Re-run `nfr-review arch --format dsl` after significant changes and compare the diff to see what shifted in the architecture.

## Drift detection (Python API)

The `structurizr_diff` module compares a baseline DSL workspace against a fresh scan to detect architectural drift:

```python
from nfr_review.structurizr_parser import parse_dsl_file
from nfr_review.structurizr_diff import diff_workspaces

baseline = parse_dsl_file("baseline.dsl")
current = parse_dsl_file("current.dsl")
report = diff_workspaces(baseline, current)

for finding in report.findings:
    print(f"[{finding.severity}] {finding.kind}: {finding.message}")
```

Drift kinds detected:

| Kind | Severity | Meaning |
|------|----------|---------|
| `element_added` | medium | New component not in baseline (undocumented growth) |
| `element_removed` | high | Component in baseline missing from scan (dead architecture) |
| `relationship_added` | medium | New dependency (unplanned coupling) |
| `relationship_removed` | medium | Dependency removed (API change) |
| `technology_changed` | low | Technology string changed (tech-stack drift) |
| `description_changed` | info | Description updated |
| `tag_changed` | low | Tags modified |
