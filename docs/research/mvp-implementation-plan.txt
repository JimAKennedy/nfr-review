# NFR Review Automation — Design and MVP Spec

*Companion to the NFR Design Review Framework and the Stack-Specific Reviewer Addendum. This document is the design starting point for the standalone automation tool. It contains the framework-to-band mapping, the MVP scope, the architecture, the rule and collector contracts, and an implementation plan ready to hand to Claude Code.*

-----

## Part 1: Framework-to-Band Mapping

### Band definitions

|Band                              |Meaning                                                                                                                                                                   |Tool output                                                                   |
|----------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------|
|**1 — Deterministic**             |Evidence is in a queryable artefact (code, IaC, config, schema registry, platform API, observability backend). A scanner produces an authoritative finding without an LLM.|Tool emits a finding with confidence “high”; reviewer typically accepts.      |
|**2 — LLM judgement on artefacts**|Artefact is structured but the question requires interpretation (ADRs, runbooks, threat models, code intent). LLM reads collector-gathered evidence and drafts a finding. |Tool emits “draft” finding with cited evidence; reviewer confirms or upgrades.|
|**3 — Hybrid (evidence-only)**    |Tool gathers and presents evidence; reviewer makes the judgement call. The value is reviewer time saved, not autonomous decision.                                         |Tool emits an “evidence pack” entry with no RAG rating; reviewer rates.       |
|**4 — Human-only**                |Genuine business, organisational, or stakeholder judgement. Tool does not attempt.                                                                                        |Reviewer handles in workshop.                                                 |

### Band totals (approximate)

- **Band 1**: ~50% of prompts
- **Band 2**: ~25%
- **Band 3**: ~15%
- **Band 4**: ~10%

Around 75% of the framework is automatable to some degree, which is the upper bound on tool coverage. The MVP targets Band 1 first because those findings are deterministic, defensible, and immediately reduce reviewer time on the most error-prone evidence.

## Part 2: MVP Scope

### Goal

A standalone Python CLI that ingests a project repository (and, optionally, references to runtime backends) and produces a draft NFR findings matrix in the existing CSV format, populated with high-confidence Band 1 findings and a small set of Band 2 LLM-assisted findings, on the highest-risk-detection prompts.

### Non-goals (explicit deferrals to later phases)

- No portfolio aggregation or web UI in MVP (Phase 3).
- No live Kafka admin / Confluent Cloud / Azure Resource Manager collectors in MVP (Phase 2). MVP works against repo + CI artefacts only.
- No Loki/Tempo/Mimir queries in MVP (Phase 2).
- No automatic ticket creation, no remediation suggestions beyond what’s in the matrix (Phase 4).
- No multi-language support — Java/Spring Boot only in MVP. Other languages can be added by writing language-specific collectors later.

### MVP scope: rules

Twenty rules, chosen for highest incident-correlation across this stack. All Band 1 except where noted. Each rule produces a finding only when the evidence supports it; absence of evidence emits a Grey finding rather than nothing.

### MVP scope: collectors

|Collector      |Inputs                                                   |Outputs                                                                                                                             |
|---------------|---------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------|
|`repo`         |Path to local git clone                                  |File index, Git metadata (CODEOWNERS, branches), language detection, dependency manifests (`pom.xml`, `build.gradle`).              |
|`java_source`  |Repo collector output                                    |AST-extracted facts: classes, methods, annotations, HTTP client usages, Kafka producer/consumer config, `@Transactional` boundaries.|
|`spring_config`|Repo collector output                                    |Parsed `application*.yml` / `application*.properties` (all profiles); resolved property tree.                                       |
|`kubernetes`   |Repo collector output                                    |Parsed manifests (Deployment, Service, PDB, ServiceAccount, secret refs).                                                           |
|`apim_policy`  |Repo collector output                                    |Parsed APIM policy XML files; rate-limit, JWT, IP filter, CORS, named-values references.                                            |

All collectors are deterministic Python. They emit JSON-serialisable evidence records to a single in-memory `EvidenceStore`.

### MVP scope: outputs

1. **Findings CSV** — exact column structure of the existing `non-functional-design-review-findings-matrix-template.csv`, ready to import into the governance workflow.
1. **Findings JSON-Lines** — one finding per line, with full evidence references, for downstream tooling and Phase 3 aggregation.
1. **Run summary** — counts by RAG, by category, by pattern tag; rules executed; rules skipped with reason; collector errors. Printed to stdout and written as `summary.json`.

### MVP success criteria

- Run end-to-end on three pilot repos (one Java microservice, one Kafka pipeline service, one mainframe-coexistence service) producing CSV output.
- For at least 60% of the rules, the tool’s findings agree with a manual reviewer’s findings on the same repo.
- False-positive rate under 20% across the pilot set.
- Run time under 2 minutes on a repo of 100k LOC.

### Data model

```python
# src/nfr_review/model.py

from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import Any


class RAG(Enum):
    GREEN = "Green"
    AMBER = "Amber"
    RED = "Red"
    GREY = "Grey"


class Severity(Enum):
    LOW = "Low"
    MEDIUM = "Medium"
    HIGH = "High"
    CRITICAL = "Critical"


@dataclass
class EvidenceRecord:
    """A single piece of evidence collected from the project."""
    collector: str            # e.g. "java_source"
    kind: str                 # e.g. "kafka_producer_config"
    locator: str              # file path + line range, or URL, or query
    data: dict[str, Any]      # collector-specific payload
    collected_at: datetime = field(default_factory=datetime.utcnow)

    @property
    def id(self) -> str:
        return f"{self.collector}:{self.kind}:{self.locator}"


class EvidenceStore:
    """In-memory store, queryable by collector+kind."""
    def __init__(self) -> None:
        self._records: list[EvidenceRecord] = []

    def add(self, record: EvidenceRecord) -> None:
        self._records.append(record)

    def query(self, *, collector: str | None = None, kind: str | None = None) -> list[EvidenceRecord]:
        results = self._records
        if collector:
            results = [r for r in results if r.collector == collector]
        if kind:
            results = [r for r in results if r.kind == kind]
        return results


@dataclass
class Finding:
    """A single review finding, ready for CSV/JSONL output."""
    rule_id: str
    primary_category: str
    secondary_category: str | None
    rag: RAG
    severity: Severity
    pattern_tag: str
    summary: str
    risk_statement: str
    recommendation: str
    evidence_ids: list[str]
    confidence: str  # "high" | "medium" | "low"
    band: int        # 1, 2, 3
    notes: str = ""


@dataclass
class RunResult:
    findings: list[Finding]
    rules_executed: list[str]
    rules_skipped: list[tuple[str, str]]  # (rule_id, reason)
    collector_errors: list[tuple[str, str]]  # (collector, error_message)
    started_at: datetime
    finished_at: datetime
```

### Rule contract

Rules are Python classes implementing a small interface. They are not data files in the MVP — Python code is more debuggable, more testable, and allows complex evidence cross-referencing. (Externalising rules to YAML is a Phase 2 option once the rule patterns stabilise.)

```python
# src/nfr_review/rules/base.py

from abc import ABC, abstractmethod
from nfr_review.model import EvidenceStore, Finding


class Rule(ABC):
    rule_id: str
    primary_category: str
    secondary_category: str | None = None
    pattern_tag: str
    band: int
    required_collectors: list[str]  # rule is skipped if any are missing

    @abstractmethod
    def evaluate(self, evidence: EvidenceStore) -> list[Finding]:
        """Return zero or more findings."""
        ...

    def applicable(self, evidence: EvidenceStore) -> bool:
        """Default: true if all required collectors produced any evidence."""
        for c in self.required_collectors:
            if not evidence.query(collector=c):
                return False
        return True
```

### Reasoner contract (Band 2)

```python
# src/nfr_review/reasoner.py

from anthropic import Anthropic
from nfr_review.model import EvidenceRecord


class Reasoner:
    def __init__(self, client: Anthropic, model: str = "claude-opus-4-7"):
        self.client = client
        self.model = model

    def judge(
        self,
        *,
        rule_id: str,
        prompt: str,
        evidence: list[EvidenceRecord],
        output_schema: dict,
    ) -> dict:
        """
        Pass curated evidence to the LLM with a strict JSON schema for output.
        Implementation uses the Anthropic Messages API with structured output.
        Each Band 2 rule has a fixed prompt template and a fixed output schema
        so behaviour is reproducible.
        """
        ...
```

The reasoner pattern is deliberately narrow: rules curate the evidence they want the LLM to look at, the LLM’s output is schema-validated, and every LLM call is logged with its inputs and outputs for audit.

### CLI

```
nfr-review run \
    --repo /path/to/project \
    --out ./review-output \
    [--rules kafka-producer-acks,resilience4j-coverage]   # filter
    [--skip-llm]                                          # disables Band 2
    [--config ./nfr-review.yaml]                          # project overrides

nfr-review list-rules
nfr-review explain <rule-id>
nfr-review version
```

`--skip-llm` is important for trial runs and offline use; the tool must produce useful output without hitting the API.

### Configuration file

A per-project `nfr-review.yaml` (committed to the repo) lets teams declare context the tool can’t infer:

```yaml
service_tier: "Tier 1"
business_criticality: "High"
data_classification: "Confidential"
application_owner: "team-payments"
review_id_prefix: "NFR-PAYMENTS"

# Hint where APIM policies live, if not auto-detectable
apim_policy_paths:
  - "infra/apim/policies/"

# Mark categories not applicable
not_applicable_categories:
  - "Migration and Coexistence"   # because this service is greenfield

# Override rules for known-accepted exceptions
rule_overrides:
  - rule_id: "kafka-consumer-autocommit"
    rag: "Green"
    reason: "Auto-commit acceptable for the metrics topic — no exactly-once requirement"
    accepted_by: "ARB-2026-Q1"
    expiry: "2026-09-30"
```

Overrides are honoured but always emit a finding with `notes` recording the override and its expiry — they don’t silently disappear from the matrix.

## Part 5: Implementation Plan

### Sequencing principle

Build vertically, not horizontally: get one rule end-to-end (collector → store → rule → CSV) before adding more rules. This proves the architecture early and lets each subsequent rule be added in isolation.

### Sprint 0: Project setup (week 1)

- Repo scaffolding, `pyproject.toml`, Python 3.12, ruff + black + mypy + pytest configured.
- Pre-commit hooks.
- GitHub Actions: lint, type-check, test on PR.
- Implement `model.py` (EvidenceStore, Finding, RunResult).
- Implement CLI skeleton with `--version` and `list-rules` working.
- One smoke test that runs the CLI and asserts non-zero output.

**Deliverable**: empty CLI runs and prints “no rules registered yet”.

### Sprint 1: First vertical slice (week 2)

- Implement `repo` collector and `java_source` collector (parse `pom.xml` and find Kafka producer config in source).
- Implement rule `kafka-producer-acks` end-to-end.
- Implement CSV writer matching the existing template exactly.
- Build the `kafka_pipeline` fixture repo with a deliberately-misconfigured producer.
- Test: run the CLI on the fixture, assert the CSV contains the expected finding.

**Deliverable**: one rule produces one finding from one fixture, output matches existing matrix format.

### Sprint 2: Kafka and resilience rules (weeks 3–4)

- Add collectors: `spring_config`, `kubernetes`.
- Add rules: `kafka-producer-idempotence`, `kafka-consumer-autocommit`, `kafka-consumer-isolation-level`, `resilience4j-coverage`, `http-client-timeout-explicit`, `aks-pdb-presence`, `aks-graceful-shutdown`.
- Expand `kafka_pipeline` and add `java_microservice` fixture.

**Deliverable**: 8 rules covering Kafka and resilience. CSV output growing.

### Sprint 3: Security, capacity, maintainability rules (weeks 5–6)

- Add collectors: `apim_policy`, `ci_artefacts`.
- Add rules: `apim-policy-jwt-presence`, `apim-named-values-keyvault`, `workload-identity-vs-sp`, `hikaricp-vs-pg-max-connections`, `java-lts-currency`, `spring-boot-support-state`, `sca-cve-exposure`, `codeowners-coverage`, `pii-in-log-statements` (Band 1 regex pre-filter only).
- Add `mainframe_adapter` fixture.

**Deliverable**: 17 deterministic rules. Run on all three fixtures producing structured output.

### Sprint 4: LLM reasoner and Band 2 rules (weeks 7–8)

- Add collectors: `adr`.
- Implement `Reasoner` against the Anthropic API with structured output.
- Add rules: `adr-implementation-drift` (Band 2), `dual-write-detection` (Band 2), upgrade `pii-in-log-statements` to Band 1+2 (regex pre-filter then LLM confirm).
- Add `--skip-llm` flag and ensure the tool produces useful output without it.
- Add LLM call logging for audit (every prompt + response saved with the run output).

**Deliverable**: 20 rules including 3 Band 2 rules. Architecture proven for both deterministic and LLM-assisted findings.

### Sprint 5: Hardening and pilot (weeks 9–10)

- JSONL output, run summary, error handling, exit codes.
- Documentation: README, rule-authoring, collector-authoring.
- Sample `nfr-review.yaml`.
- Pilot on three real internal projects.
- Capture detection rate, false-positive rate, runtime.

**Deliverable**: MVP ready for wider pilot. Pilot report drives Phase 2 priorities.

- Solo developer with Claude Code: 8–10 calendar weeks.
- Two developers in parallel: 5–6 weeks (collectors and rules parallelise well).
- Faster if existing internal tooling can be reused for some collectors (e.g. existing SCA report parsers, existing APIM policy linters).

## Part 6: Notes for Claude Code

These are pointers for the AI agent that will build this — they bias toward your stated patterns and away from common LLM-tool failure modes.

- Build vertically. One rule fully working end-to-end before adding the second. Resist the urge to scaffold all 20 rules at once; the architecture changes during the first one.
- Collectors are pure functions of inputs to evidence. No hidden state, no global config, no network calls (except the future `apim_runtime`/`kafka_admin` collectors in Phase 2). This makes them trivially testable.
- Every rule has at least three tests: a fixture where the rule fires, a fixture where it doesn’t, and an edge case (typically: required evidence missing, expected to emit Grey or skip).
- Use `tree-sitter` or `javalang` for Java AST parsing rather than regex. Regex on Java source is the path to brittleness; the small upfront cost of a real parser pays off by rule three.
- Use `lxml` for APIM policy XML, not regex.
- Use `ruamel.yaml` (round-trip safe) for Spring config, not stdlib `yaml`.
- The Reasoner should never be passed an entire repo. Each Band 2 rule curates a small, named evidence bundle (typically under 8KB) and asks a narrow question. If a rule needs more, split it.
- Structured output for the Reasoner: use the Anthropic API with a JSON schema, validate the response, retry once on schema violation, fail the rule with a clear error if it can’t be parsed.
- Pattern tags are first-class: every rule has one (or, rarely, declares a new one in the rule code with a registry entry). The tag list lives in `rules/registry.py` and is the canonical taxonomy for the project.
- Keep the CSV writer faithful to the existing template column order and types. The whole point of the tool is that its output drops into the existing governance workflow without translation.
- Ship a `SKILL.md` at `docs/skill.md` so other Claude Code or OpenCode projects can invoke this tool through their agent. The skill describes the CLI, the inputs it expects, and how to interpret the CSV.
