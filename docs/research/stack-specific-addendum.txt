# Non-Functional Design Review Framework — Stack-Specific Reviewer Addendum

*Java microservices · LGTM · Confluent Kafka · Azure APIM · Mainframe modernisation · External SaaS*

## 1. Purpose and Use

This addendum extends the Non-Functional Design Review Category Guide with stack-specific reviewer prompts, common risks, and pattern tags. It is not a replacement for the canonical category guide; it is a companion that sharpens reviewer probing in environments dominated by Java microservices on AKS, Confluent Kafka, Azure API Management, observability via the LGTM stack (Loki, Grafana, Tempo, Mimir), mainframe modernisation, and integration with external SaaS providers.

Reviewers should continue to assign findings to the canonical primary categories (Architecture Fitness, Integration and API Design, Resilience and Dependency Failure, and so on). The prompts below are additional probes within those categories — not new categories themselves — except for the new cross-cutting Migration and Coexistence area, which is recommended as either a dedicated category or a mandatory cross-cutting lens during transitional states.

The intent of every addition is to prevent the most common Green-rating-but-actually-Red outcome in this stack: a system that passes generic NFR prompts because the prompts are too high-level to detect the specific misconfigurations that cause incidents in production.

### How to use this document

- Use it alongside the canonical category guide during the evidence review and review workshop stages.
- For each canonical category, treat the prompts here as a supplementary checklist. They are written assuming the canonical questions have already been answered.
- Pattern tags introduced here should be added to the controlled tag vocabulary maintained by the architecture governance lead. Do not invent ad-hoc tags during a review; either use one from this document or propose a new tag through the read-across owner.
- If the application under review does not use a particular component (for example, no mainframe coexistence), mark that section Grey rather than skipping it silently — it preserves portfolio comparability.

### Relationship to the canonical category guide

The canonical guide describes the review intent and stack-agnostic questions for each category. This addendum adds:

- Specific configuration and behavior probes for the technologies in scope.
- Failure modes that are common in this stack and rarely surfaced by generic prompts.
- Pattern tags shaped to enable portfolio read-across across multiple Java/Kafka/APIM applications.
- One new cross-cutting category — Migration and Coexistence — that does not have a clean home in the existing guide.

## 2. Architecture Fitness and Standard Patterns (Stack Probes)

#### Intent

Confirm that the architecture is intentionally designed for a Java/AKS/Kafka/APIM environment with mainframe coexistence, and that pattern choices reflect the operational and integration realities of that environment rather than copying generic microservice templates.

#### Reviewer prompts

|Prompt area                  |Reviewer should probe                                                                                                                                                                                                     |
|-----------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|Service decomposition        |Are bounded contexts aligned to data ownership boundaries that survive the mainframe migration, or are they shaped around current mainframe transaction boundaries that will themselves change?                           |
|Sync vs async per dependency |Has each dependency edge been classified as sync HTTP via APIM, async event via Kafka, batch file, or mainframe-attached, with explicit rationale captured in an ADR?                                                     |
|Pattern semantics, not syntax|Where Kafka is used, is it used as a durable log with replay and schema governance, or merely as an asynchronous queue? Where APIM is used, is it a security and policy boundary, or only a routing convenience?          |
|Platform fit                 |Are AKS workloads deployed in line with the enterprise landing zone (node pools, network policies, ingress, identity binding), or do they assume a generic Kubernetes baseline?                                           |
|Codebase to architecture     |Does the repository structure reflect the bounded contexts in the architecture, or does the implementation reveal hidden cross-context coupling through shared libraries, shared database schemas, or shared Kafka topics?|

#### Common risks

- Bounded contexts shaped around current mainframe transaction layouts that will be redesigned during migration, forcing rework downstream.
- Kafka treated as ‘just a queue’ with topics designed for current consumers only, no schema governance, and no replay design.
- APIM treated as routing rather than a security boundary, leading to policies (rate-limit, JWT validation, IP filtering) being bypassed when traffic enters from internal networks or self-hosted gateways.
- Shared Java libraries that hard-couple unrelated services through transitive dependencies, breaking the pattern of independent deployability.
- ADRs that describe the intent but contradict the deployed AKS manifests, APIM policies, or Kafka topic configuration.

#### Pattern tags for read-across

|Pattern tag                 |Use when                                                        |
|----------------------------|----------------------------------------------------------------|
|`bounded-context-misaligned`|Bounded context boundaries do not match data ownership.         |
|`kafka-used-as-queue`       |Kafka used without log, replay, or schema governance properties.|
|`apim-as-router-only`       |APIM in path but not enforcing security or policy.              |
|`shared-lib-coupling`       |Cross-service coupling through shared Java libraries.           |
|`adr-implementation-drift`  |ADR present but contradicts deployed configuration.             |

## 3. Integration and API Design (Stack Probes)

#### Intent

Confirm that integration choices across APIM, Kafka, mainframe adapters, and SaaS endpoints are explicit, evolvable, and operationally supportable. Generic prompts about idempotency and ordering are not enough; this stack has specific configuration surfaces that must be inspected.

#### Reviewer prompts

|Prompt area                    |Reviewer should probe                                                                                                                                                                                                                                                            |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|APIM policy correctness        |For each APIM API and product, are rate-limit, quota, JWT validation, IP filtering, body transformation, and CORS policies actually configured at the right scope (global, product, API, operation), and tested? Are named values backed by Key Vault rather than inline secrets?|
|APIM trust boundary            |Is APIM the enforced entry point for all external traffic, or are there bypass paths via internal networks, self-hosted gateways, or direct AKS ingress? Are self-hosted gateway placements (for example near mainframe-adjacent traffic) reviewed and justified?                |
|APIM subscription and key model|Are subscription keys rotated, scoped per consumer, and revocable? Is there a tested process for emergency revocation?                                                                                                                                                           |
|Trace context propagation      |Does the W3C traceparent header survive the full path APIM → service → Kafka headers → consumer service → mainframe adapter → SaaS callback? Is it preserved by APIM policies (a common loss point) and propagated as Kafka headers by producers?                                |
|Kafka producer guarantees      |What are the actual values of acks, enable.idempotence, max.in.flight.requests.per.connection, and transactional.id for each producer? Do the values support the durability and ordering claims made in the design?                                                              |
|Kafka consumer guarantees      |Is auto-commit disabled where exactly-once or at-least-once semantics are claimed? Is offset commit explicit and tied to successful side-effect completion? Is isolation.level set to read_committed where transactional producers feed the topic?                               |
|Schema Registry governance     |What is the compatibility mode per subject (BACKWARD, FORWARD, FULL, FULL_TRANSITIVE, NONE)? Who approves schema changes? Are breaking changes detectable in CI? Are subject naming conventions consistent (TopicNameStrategy vs RecordNameStrategy vs TopicRecordNameStrategy)? |
|Partition key strategy         |What is the partition key for each topic, and does it produce a balanced distribution under realistic traffic? Has hot-partition risk been measured, not just modeled? Does the key choice preserve the ordering guarantee that consumers actually rely on?                      |
|DLQ and poison message handling|Is there a DLQ topic per consumer group with a documented retention, replay tool, and alerting on inflow rate? Is the DLQ schema-compatible with the source topic, or does it lose data on serialisation failure?                                                                |
|Outbox vs dual-write           |Where a service writes to a database and publishes to Kafka, is the transactional outbox pattern in use (with Debezium or equivalent CDC), or is the service performing a dual-write that can lose events on partial failure?                                                    |
|SaaS webhook ingress           |For each SaaS webhook, what is the delivery guarantee (at-least-once, at-most-once)? Is there idempotent processing keyed on a vendor-supplied delivery ID? Is the replay window sufficient to recover from a multi-hour outage on the consumer side?                            |
|SaaS API egress                |For each SaaS API call, what are the rate limits and the consequences of breaching them (throttling, blocking, contractual)? Is backoff implemented, and is the backoff cap below the SaaS provider’s penalty threshold?                                                         |
|OAuth and token lifecycle      |How are OAuth client credentials stored, rotated, and refreshed for each SaaS? What happens to in-flight requests when the SaaS identity provider is unavailable for token refresh?                                                                                              |

#### Common risks

- APIM rate-limit or JWT policies set at the wrong scope, so they apply to some operations and silently not to others.
- Trace context lost at APIM because the policy chain rewrites or strips headers, breaking end-to-end observability.
- Kafka producer claims exactly-once but max.in.flight.requests.per.connection > 5 with retries enabled, breaking ordering on retry.
- Schema Registry compatibility mode set to NONE on critical topics, allowing breaking changes to ship undetected.
- Partition key based on a low-cardinality field (for example tenant ID with one dominant tenant), creating a hot partition that bottlenecks throughput.
- DLQ exists but has no consumer, no alert on inflow, and no replay tool — poison messages accumulate unnoticed for months.
- Service writes to Postgres and produces to Kafka in the same method without an outbox, losing events on JVM crash between the two operations.
- SaaS webhook handler is not idempotent; vendor retries cause duplicate processing on every transient failure.
- OAuth tokens cached in memory only; a restart during a SaaS IdP outage causes total dependency failure rather than degraded operation.

#### Pattern tags for read-across

|Pattern tag                  |Use when                                                         |
|-----------------------------|-----------------------------------------------------------------|
|`apim-policy-scope-error`    |APIM policy applied at wrong scope or missing on some operations.|
|`apim-bypass-path`           |Traffic path that bypasses APIM enforcement.                     |
|`traceparent-broken`         |W3C trace context lost across a hop.                             |
|`kafka-eos-claim-unverified` |Exactly-once claimed but configuration does not support it.      |
|`kafka-acks-mismatch`        |Producer acks setting inconsistent with durability claim.        |
|`schema-registry-compat-weak`|Compatibility mode permits breaking changes.                     |
|`hot-partition-risk`         |Partition key likely to produce uneven distribution.             |
|`dlq-unmonitored`            |Dead-letter topic with no alerting or replay.                    |
|`dual-write-anti-pattern`    |DB write and Kafka produce without outbox or CDC.                |
|`webhook-not-idempotent`     |Webhook handler not safe under vendor retry.                     |
|`saas-rate-limit-untested`   |SaaS rate limit not modeled or tested.                           |
|`oauth-refresh-fragile`      |Token refresh path fails on IdP outage with no degraded mode.    |

## 4. Resilience and Dependency Failure (Stack Probes)

#### Intent

Confirm that resilience patterns are configured correctly in the actual code and infrastructure, not merely named in the design. Java services tend to claim Resilience4j or similar without wiring it into the right call paths, and Kafka-based designs frequently lack failure-mode capacity testing.

#### Reviewer prompts

|Prompt area                        |Reviewer should probe                                                                                                                                                                                                                                                            |
|-----------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|Resilience4j wiring (or equivalent)|For each external dependency, is there an actual Resilience4j (or equivalent) circuit breaker, retry, time limiter, and bulkhead annotation or programmatic wrap? Are the configuration values explicit, externalised, and tuned per dependency rather than copied from a sample?|
|Timeout layering                   |Are connect, read, and overall request timeouts set explicitly at the HTTP client (for example WebClient, RestTemplate, OkHttp), and do they sum to a value below the upstream APIM timeout and the user SLO?                                                                    |
|Retry safety                       |Are retries restricted to idempotent operations? Is retry budget bounded (max attempts and total time)? Is jitter applied? Is the retry observable in logs and metrics?                                                                                                          |
|Kafka consumer failure modes       |What happens when the consumer fails to process a message: retry with backoff, send to DLQ after N attempts, or block the partition? Is the choice documented per topic? Has the behaviour been tested under poison-message conditions?                                          |
|Kafka cluster failure              |Has the service been tested with min.insync.replicas not satisfied (broker outage)? Does the producer block, fail fast, or buffer? Is the buffer bounded and observable?                                                                                                         |
|AKS pod and node failure           |Are PodDisruptionBudgets configured? Are readiness and liveness probes distinct and meaningful? Does the service handle SIGTERM gracefully, draining in-flight requests and committing Kafka offsets before exit?                                                                |
|SaaS dependency failure modes      |For each SaaS dependency, what is the documented degraded mode? Stale cache served with caveat? Queue and retry? Hard fail? Has the user-visible behaviour been reviewed by product?                                                                                             |
|Mainframe-attached failure modes   |When the mainframe adapter (CICS, MQ, file transfer) is slow, throttling, or unavailable, what does the cloud service do? Is there a circuit breaker on that path? Is there a fallback to cached state, and how stale is acceptable?                                             |
|DR and failover testing            |Have the AKS multi-region (or zone-redundant) failover, Kafka MirrorMaker / Cluster Linking failover, and APIM regional failover been tested in a drill within the last twelve months? Is there evidence of the test, not just a runbook?                                        |
|Backup and restore evidence        |For Postgres and any other stateful store, has a restore been performed end-to-end with production-volume data within the last six months? Is the restore tied to an RPO/RTO that the business has signed off on?                                                                |

#### Common risks

- Resilience4j present in pom.xml but only annotated on a subset of clients, leaving the riskiest paths unprotected.
- Timeouts set at the Spring properties level but overridden by a downstream HTTP client default, so the effective timeout is the platform default.
- Retries enabled at multiple layers (client library, Spring Retry, manual code) producing exponential retry storms during partial outages.
- Kafka consumer with no DLQ and no max retry, so a poison message blocks the partition indefinitely.
- SIGTERM not handled; pod terminations during deploys cause uncommitted offsets and duplicate processing on restart.
- Mainframe adapter has no circuit breaker; mainframe slowness propagates as full-stack thread exhaustion within minutes.
- DR runbook exists but the documented failover procedure has never been executed, and the most recent change to the runbook is over two years old.
- Postgres backup runs nightly but the most recent restore drill is older than the most recent schema migration, so restore is unverified against current schema.

#### Pattern tags for read-across

|Pattern tag                    |Use when                                                 |
|-------------------------------|---------------------------------------------------------|
|`resilience4j-partial-coverage`|Resilience patterns missing on subset of dependencies.   |
|`timeout-not-explicit`         |Timeout relies on platform default.                      |
|`retry-storm-risk`             |Retries layered without coordination.                    |
|`dlq-not-configured`           |Consumer has no DLQ path.                                |
|`sigterm-not-handled`          |Graceful shutdown not implemented.                       |
|`mainframe-no-breaker`         |No circuit breaker on mainframe-attached path.           |
|`dr-untested`                  |DR runbook not exercised within agreed cadence.          |
|`restore-unverified`           |Backup taken but restore not drilled with current schema.|

## 5. Performance and Latency (Stack Probes)

#### Intent

Confirm that performance is modelled and measured against the specific layers of this stack: JVM behaviour, APIM policy overhead, Kafka end-to-end lag, and mainframe-attached latency.

#### Reviewer prompts

|Prompt area               |Reviewer should probe                                                                                                                                                                                                               |
|--------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|JVM tuning                |Which garbage collector is in use (G1, ZGC, Shenandoah, Parallel)? Are GC pause-time SLOs defined? Are heap, metaspace, and direct memory sized against measured workload, not Spring Boot defaults?                                |
|Virtual threads (Java 21+)|If virtual threads are in use, has pinning behaviour been audited (synchronized blocks, native calls, JNI)? Is monitoring in place for carrier thread starvation?                                                                   |
|Thread pool sizing        |For services performing blocking I/O, are thread pools sized using Little’s Law against measured downstream latency, or with arbitrary defaults? Is there separation between request-handling, Kafka consumer, and background pools?|
|Connection pool sizing    |Are HikariCP (Postgres) and HTTP client pool sizes consistent with thread pools and downstream capacity? Is pool exhaustion alerted on?                                                                                             |
|APIM policy overhead      |Has the latency cost of each APIM policy been measured? Are heavy policies (body transforms, external auth callouts) on hot paths, and is that visible in Tempo traces?                                                             |
|Kafka end-to-end latency  |Is end-to-end latency (produce time to consume time) measured per topic, not just produce or consume separately? Is there an SLO on consumer lag, and is it driven by user impact rather than a generic threshold?                  |
|Mainframe-attached latency|What is the measured p50, p95, p99 latency for each mainframe call, and what is the historical variance? Is the latency budget for the user journey set against the worst observed (not average) mainframe response?                |
|Performance test realism  |Do load tests include APIM, JWT validation, encryption, logging, and mainframe-attached calls — or do they bypass them with stubs that misrepresent the production path? Is test data volume representative of production?          |
|Tail latency reporting    |Are p95, p99, and p999 percentiles reported per endpoint and per Kafka topic, not just averages? Are tail latency SLOs defined for user-facing journeys?                                                                            |
|Cache effectiveness       |Where caches (Redis, in-process Caffeine, APIM cache) are used, is hit rate measured and alerted on? Is cache stampede prevention in place for high-traffic keys?                                                                   |

#### Common risks

- Default Spring Boot Tomcat thread pool of 200 used for services that depend on a 2-second mainframe call, causing thread exhaustion at modest concurrency.
- Virtual threads adopted without auditing for pinning, causing carrier thread starvation under load on any synchronized path.
- HikariCP pool sized larger than the Postgres max_connections divided by replica count, allowing the application to overwhelm the database.
- APIM body transformation policy on hot paths adds 30–80 ms per request and is invisible because the trace span is missing.
- Kafka consumer lag alert set on absolute lag rather than time lag, so a slow consumer is invisible until lag is enormous.
- Mainframe latency budgeted at the average; production behaviour during batch windows breaches user SLO repeatedly.
- Load test bypasses APIM and uses HTTP stubs for the mainframe, so the test result has no relationship to production behaviour.
- Only average response time reported; p99 latency has been over the SLO for months and nobody noticed.

#### Pattern tags for read-across

|Pattern tag                  |Use when                                                |
|-----------------------------|--------------------------------------------------------|
|`gc-untuned`                 |GC settings unreviewed against workload.                |
|`virtual-thread-pinning-risk`|Virtual threads adopted without pinning audit.          |
|`thread-pool-default`        |Thread pool sized at framework default.                 |
|`pool-larger-than-db`        |Connection pool exceeds DB capacity.                    |
|`apim-policy-hot-path`       |Heavy APIM policy on latency-critical operation.        |
|`lag-alert-on-count`         |Consumer lag measured in messages, not time.            |
|`mainframe-budget-on-mean`   |Mainframe latency budget assumes mean, not tail.        |
|`loadtest-bypasses-prod-path`|Load test bypasses APIM, auth, or critical dependencies.|
|`tail-latency-not-reported`  |p99 not measured or reported.                           |

## 6. Capacity and Scalability (Stack Probes)

#### Intent

Confirm that capacity is modelled against the actual scaling units of this stack: AKS pods and nodes, Kafka partitions and consumer instances, APIM tier limits, Postgres connections, and SaaS rate limits.

#### Reviewer prompts

|Prompt area                     |Reviewer should probe                                                                                                                                                                                                                                                                           |
|--------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|AKS scaling design              |Is HPA configured on user-impact signals (RPS, queue depth, p95 latency) or on CPU alone? Is the cluster autoscaler configured with sensible node pool boundaries? Have warm-up times been measured and accounted for in scale-up decisions?                                                    |
|Kafka partition planning        |Is partition count sized for peak parallelism, with headroom for future growth? Has the operational cost of partition count (per-broker file handles, leader election time, controller load) been considered? Is there a process for adding partitions, given that it breaks key-based ordering?|
|Kafka consumer parallelism      |Does consumer instance count match partition count for the consumer group? Is rebalancing cost measured? Is cooperative-sticky assignment in use to reduce rebalance impact?                                                                                                                    |
|Kafka tiered storage            |If tiered storage is enabled (Confluent), are retention policies aligned to replay needs? Is the cost of object storage versus broker storage understood per topic?                                                                                                                             |
|APIM tier and limits            |Is the APIM tier (Developer, Basic, Standard, Standard v2, Premium, Premium v2) appropriate for throughput, latency, and HA needs? Are throttling policies sized against the tier’s actual capacity, not its marketing claims?                                                                  |
|Postgres capacity               |Is max_connections sized against the sum of all application pool sizes plus admin headroom? Is connection pooling via PgBouncer or similar in place where pool sums exceed comfortable limits? Are noisy queries, table bloat, and replication lag monitored?                                   |
|SaaS rate limit headroom        |For each SaaS dependency, what is the rate limit and what is the current peak utilisation? Is there a margin for traffic spikes, retries during partial outages, and growth?                                                                                                                    |
|Failure-mode capacity           |When a downstream is degraded, retries multiply request volume. Has capacity been modelled for the retry-amplified traffic shape, not just steady-state?                                                                                                                                        |
|Batch and reconciliation traffic|Are batch jobs, reconciliation runs, reporting queries, and analytics ETL accounted for in capacity planning, or do they share infrastructure with user-facing workloads without isolation?                                                                                                     |
|Mainframe MIPS impact           |For mainframe-attached operations during coexistence, has the additional MIPS cost from cloud-side traffic been measured and budgeted? Is there an alert when cloud-side volume drives mainframe utilisation above contracted thresholds?                                                       |

#### Common risks

- HPA on CPU only; service scales late because the bottleneck is downstream latency, not CPU.
- Kafka topic provisioned with 100 partitions for future scale, but consumer group has 4 instances and 96 partitions are idle while 4 are saturated.
- Adding partitions to fix throughput silently breaks key-based ordering for downstream consumers.
- APIM Standard tier hits documented unit cap during marketing campaigns; throttling kicks in as user-visible 429s.
- Postgres max_connections is 100, four service replicas each have a HikariCP pool of 30, plus three more services share the database — pool exhaustion is a question of when, not if.
- SaaS rate limit utilisation is 80% at peak; any retry storm pushes the system over the contractual limit.
- Capacity model assumes steady-state traffic; during the last incident, retry amplification produced 3x normal traffic and saturated everything downstream.
- Mainframe MIPS spike during cloud-side load testing produces a finance escalation that the team did not anticipate.

#### Pattern tags for read-across

|Pattern tag                     |Use when                                            |
|--------------------------------|----------------------------------------------------|
|`hpa-cpu-only`                  |Autoscaling on CPU rather than user-impact signal.  |
|`partition-count-mismatch`      |Partition count not aligned to consumer parallelism.|
|`partition-add-breaks-ordering` |Plan to add partitions without ordering migration.  |
|`apim-tier-undersized`          |APIM tier inadequate for peak.                      |
|`pg-max-conn-undersized`        |Postgres max_connections below total pool capacity. |
|`saas-headroom-thin`            |SaaS rate limit close to current peak.              |
|`retry-amplification-unmodelled`|Capacity model omits retry amplification.           |
|`mainframe-mips-unbudgeted`     |Cloud-side load can drive unbudgeted MIPS.          |

## 7. Security and Privacy (Stack Probes)

#### Intent

Confirm that security controls are configured correctly across the specific surfaces of this stack: APIM as the policy enforcement point, Kafka ACLs and encryption, AKS workload identity, mainframe identity bridging, and SaaS credential management.

#### Reviewer prompts

|Prompt area                    |Reviewer should probe                                                                                                                                                                                                          |
|-------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|APIM as enforcement point      |Are JWT validation, IP filtering, mTLS, and rate-limit policies enforced at APIM, or are they assumed to be enforced and not? Are the JWKS endpoints, issuers, and audiences correct per environment?                          |
|mTLS coverage                  |Where mTLS is claimed between services (or between APIM and AKS), are the certificates actually validated, the CN/SAN checked, and the certificates rotated automatically? Is there an alert on imminent expiry?               |
|Workload identity              |Are AKS workloads using Azure AD Workload Identity (or equivalent) rather than long-lived service principal credentials? Are RBAC scopes least-privilege per workload?                                                         |
|Kafka ACLs and authentication  |Is each producer and consumer authenticated (SASL/OAUTHBEARER, mTLS) with topic-level ACLs? Is the authentication mechanism consistent or a mix of legacy and modern that creates exception paths?                             |
|Schema Registry security       |Who can register or modify schemas? Is the registry protected by the same identity model as the brokers?                                                                                                                       |
|Secrets management             |Are secrets (DB passwords, SaaS API keys, OAuth client secrets, certificates) stored in Key Vault and consumed via CSI driver, Workload Identity, or APIM named values, never in environment variables baked into images?      |
|Mainframe identity bridging    |How is identity bridged between Entra ID (cloud) and RACF/ACF2/Top Secret (mainframe)? Is the bridging tested, audited, and revocation-fast?                                                                                   |
|SaaS credential lifecycle      |For each SaaS dependency, what is the credential rotation cadence? Is rotation automated or manual? Is there an alert on credential age?                                                                                       |
|Sensitive data in observability|Are PII and secrets prevented from entering Loki logs, Tempo traces, and Mimir metrics? Are there structured logging policies and DLP scanning?                                                                                |
|Kafka payload protection       |Is payload encryption (in addition to TLS in transit) in use for sensitive topics, or is broker-level access deemed sufficient? Is the choice justified by data classification?                                                |
|DLQ data sensitivity           |Do DLQ topics inherit the data classification of their source? Are DLQ access controls equivalent? Are retention periods compatible with privacy obligations?                                                                  |
|Threat model currency          |Is the threat model updated to cover the actual deployed topology (APIM ingress, AKS, Kafka, Postgres, SaaS, mainframe), not the original design? Has it been revisited since the most recent significant architectural change?|

#### Common risks

- APIM JWT validation policy uses the wrong issuer in non-production, succeeds in tests, and fails open in production.
- mTLS between services is configured but the certificate validation is set to permissive mode for ‘compatibility’, accepting any certificate.
- AKS workload uses a long-lived service principal in a Kubernetes secret because Workload Identity migration was deferred.
- Kafka cluster has a mix of SASL/PLAIN (legacy) and SASL/OAUTHBEARER (modern) consumers with the legacy path unrestricted.
- Postgres password in a Kubernetes Secret rotated quarterly by a manual ticket, with no automated revocation when staff change.
- Identity bridging between Entra ID and RACF is implemented as a nightly batch sync, leaving up to 24 hours of stale access after revocation.
- OAuth client secret for a critical SaaS has not been rotated in over two years; the rotation runbook references a person who has left.
- PII appears in stack traces logged to Loki because exception handling logs the full request payload on failure.
- DLQ topic inherits topic ACLs but not data sensitivity controls; engineers replay DLQ messages from local laptops.
- Threat model dates from initial design; APIM, Kafka, and SaaS additions never went through threat-model update.

#### Pattern tags for read-across

|Pattern tag             |Use when                                                      |
|------------------------|--------------------------------------------------------------|
|`apim-policy-env-drift` |APIM policy values differ between environments in unsafe ways.|
|`mtls-permissive`       |mTLS configured but validation relaxed.                       |
|`sp-secret-long-lived`  |Long-lived service principal credentials in use.              |
|`kafka-mixed-auth`      |Mixed Kafka authentication mechanisms with weak fallback.     |
|`secret-rotation-manual`|Secret rotation depends on manual process.                    |
|`mainframe-id-stale`    |Identity bridge to mainframe has unsafe lag.                  |
|`saas-credential-stale` |SaaS credential not rotated within policy window.             |
|`pii-in-observability`  |Sensitive data leaking into logs, traces, or metrics.         |
|`dlq-access-divergent`  |DLQ access controls not equivalent to source topic.           |
|`threat-model-stale`    |Threat model not updated for current topology.                |

## 8. Data Management and Integrity (Stack Probes)

#### Intent

Confirm that data ownership, evolution, reconciliation, and recovery are designed for an event-driven, multi-store environment with mainframe coexistence. The canonical category covers the principles; this section covers the surfaces where they fail in practice.

#### Reviewer prompts

|Prompt area                   |Reviewer should probe                                                                                                                                                                                                                                    |
|------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|System of record clarity      |For each critical entity (customer, account, balance, order), which system is authoritative during the migration: mainframe, Postgres, Kafka log, or SaaS? Is the answer documented, and does it change over the migration timeline?                     |
|CDC ordering and lag          |For CDC pipelines (Debezium, Qlik Replicate, Connect CDC, DMS) feeding Kafka, what is the measured lag, and is ordering preserved per primary key? What happens during connector failover?                                                               |
|Outbox pattern verification   |Where the outbox pattern is claimed, is the outbox table actually consumed by CDC, and is the consumer ordered, idempotent, and monitored? Is the outbox table cleaned up?                                                                               |
|Compacted topics              |For compacted topics holding state, are tombstones produced correctly on deletion? Is compaction lag monitored? Is cleanup.policy actually compact rather than delete-and-compact misconfiguration?                                                      |
|Schema evolution governance   |How are breaking changes handled across producer and consumer release cycles? Is there a deprecation process? Are old schema versions retained long enough for slow consumers to upgrade?                                                                |
|Reconciliation controls       |For dual-state systems (mainframe + cloud, source-of-truth + materialised view), is there an automated reconciliation job that detects divergence within a defined window? Are divergences alerted, triaged, and resolved?                               |
|EBCDIC and packed-decimal     |For mainframe-sourced data, is character set conversion (EBCDIC to UTF-8) and packed-decimal conversion correct for edge cases (negative zero, signed overpunch, mixed-case national characters)? Is conversion tested against production-realistic data?|
|Backup and restore drills     |For Postgres and any other stateful store, when was the most recent restore drill, with what data volume, and was the restored system functionally tested? Is the RPO/RTO supported by the evidence, not just documented?                                |

#### Common risks

- Two systems both consider themselves authoritative for customer balance during migration; reconciliation runs nightly, exposing customers to inconsistent reads during the day.
- Debezium connector restart causes a full snapshot, producing duplicate events that downstream consumers were not designed to handle idempotently.
- Outbox pattern documented but the cleanup job has been disabled for a year; the table is now hundreds of GB and slowing inserts.
- Compacted topic intended to hold latest-value state has cleanup.policy set to delete, so old values disappear after retention regardless of compaction.
- Schema change deployed as backward-incompatible; slow consumers (especially those reading from earlier offsets) fail silently with deserialisation errors that go to DLQ.

#### Intent

#### Common risks

- Migration cutover rollback is documented but never exercised; the first rollback attempt during go-live takes hours.
- Flaky test ignored for six months; when it finally surfaces a real bug, no one believes it.

## 15. Stack-Aware Severity Rubric

To reduce drift across reviewers, this rubric refines the canonical RAG ratings for the additions in this addendum. It does not replace the canonical RAG criteria; it disambiguates them for stack-specific findings.

|Rating   |Stack-aware criteria                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
|---------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|**Red**  |Configuration or behaviour likely to cause a P1 or P2 incident under realistic load or failure conditions, or breach a regulatory obligation, or produce a data integrity issue that is hard to detect or recover. Examples: dual-write without outbox, schema-break shipped to production, Resilience4j missing on a critical dependency, traceparent dropped at APIM with no end-to-end correlation, EBCDIC conversion incorrect for edge cases, rollback path unverified at cutover.|
|**Amber**|Material gap with documented owner, mitigation, and target date. Examples: APIM tier appropriate today but headroom thin against forecast, schema compatibility check disabled in CI for a low-risk topic with manual review, mainframe MIPS impact not forecast but tracked, observability sampling rate likely insufficient but with an in-flight remediation.                                                                                                                       |
|**Green**|Configuration verified against code or deployed state, evidence current, behaviour exercised in test or production, and consistent with the canonical Green criteria. Stack-specific Green requires evidence specific to the stack, not generic claims (for example, ‘circuit breakers in place’ is not evidence; the Resilience4j configuration values, the wrapped methods, and a recent failure-mode test are).                                                                     |
|**Grey** |Not applicable (component not in use), not assessed (out of scope for this review), or insufficient evidence. Use Grey rather than skipping; it preserves portfolio comparability.                                                                                                                                                                                                                                                                                                     |

## 16. Read-Across Use of Pattern Tags

Each section above introduces pattern tags shaped to the stack. These tags should be added to the controlled vocabulary maintained by the architecture governance lead and used consistently across reviews. The intended portfolio outcomes are:

- Identify systemic gaps that recur across applications (for example, `traceparent-dropped-at-apim` appearing on multiple services suggests a platform-level fix, not multiple application-level fixes).
- Track remediation effectiveness by counting tag recurrence over time.
- Inform enterprise standards updates when a tag appears repeatedly enough to indicate the standard is the wrong shape, not the application.
- Direct platform investment toward shared concerns (APIM policy templates, Resilience4j shared library, observability conventions) rather than per-team remediation.

## 17. Lightweight Review Mode for Post-Incident Use

The canonical framework lists post-incident review as an entry point but defines a single evidence pack that may be impractical immediately after an incident. The following lightweight evidence pack is sufficient for a post-incident NFR review, with the expectation that any Red findings will be reconfirmed against the full evidence pack within ten working days.

- Incident timeline and post-incident review document.
- Logs, traces, and metrics covering the incident window.
- Configuration of the components implicated in the incident (APIM policies, Resilience4j config, Kafka topic and consumer group, AKS deployment manifests).
- Most recent change to those components (commit, pipeline, change ticket).
- The relevant runbook used during the incident, with notes on what worked and what did not.
- Any prior reviews that touched the same components, for trend analysis.

## 18. Review Effectiveness Measurement

To improve the framework over time, the architecture governance function should track:

- Closure rate by category and by RAG rating.
- Mean time to remediate, segmented by category and severity.
- Tag recurrence: how often the same pattern tag appears across reviews.
- False-positive rate: findings reclassified Green on re-rating, by category.
- Grey-to-rated conversion rate: how many Grey findings are evidenced and rated within the agreed window.
- Incident correlation: how many production incidents had a related Red or Amber finding in a recent NFR review (under-detection signal).

These metrics close the loop and let the framework itself be improved evidence-led, in the same posture it asks of the applications it reviews.

## 19. Recommended Adoption Path

- Treat this addendum as a draft for review by the architecture governance function and a representative review chair before formal adoption.
- Pilot it on two or three reviews covering different parts of the stack (for example a Java microservice, a Kafka-heavy event pipeline, and a mainframe-coexistence workflow) and gather reviewer feedback.
- Add the pattern tag taxonomy to the controlled vocabulary and brief reviewers.
- Update the findings matrix template to include a Migration and Coexistence column where applicable.
- Set a six-month review point to refresh prompts as the platform evolves (for example as Confluent, APIM, AKS, and LGTM features change).
