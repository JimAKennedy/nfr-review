# Non-Functional Design Review Category Guide

## How to Use This Guide

This guide describes the review intent, evidence expectations, common risks, and recommended reading for each non-functional review category. It is aimed at application owners, engineering leads, architects, testers, service owners, and governance reviewers. The categories intentionally overlap where real systems overlap; reviewers should assign each finding to the primary category that best drives ownership, then use secondary category and read-across tags for cross-cutting themes.

## Category Summary

| Category | Primary Question | Typical Evidence |
| --- | --- | --- |
| Architecture Fitness and Standard Patterns | Is the design coherent, supportable, and aligned to enterprise-approved patterns? | Architecture diagrams, ADRs, reference architecture mapping, codebase structure, platform standards evidence. |
| Integration and API Design | Are service interactions explicit, reliable, evolvable, and appropriate for latency and consistency needs? | API specs, event schemas, dependency diagrams, message broker config, contract tests, timeout and idempotency design. |
| Resilience and Dependency Failure | Does the service continue safely, degrade gracefully, and recover predictably when dependencies fail? | Failure mode analysis, retry and circuit breaker config, failover tests, DR tests, incident history, chaos test results. |
| Performance and Latency | Can the service meet response-time and throughput expectations under realistic load? | Performance test reports, APM traces, production latency percentiles, query plans, resource metrics. |
| Capacity and Scalability | Can the platform handle expected growth, spikes, queues, and resource saturation? | Workload model, capacity plan, autoscaling rules, load tests, queue depth trends, quota and limit analysis. |
| Security and Privacy | Are confidentiality, integrity, availability, and privacy protected by design and verified in delivery? | Threat model, control mapping, SAST/DAST, dependency scans, access review, encryption and secrets evidence. |
| Data Management and Integrity | Is data owned, protected, retained, reconciled, recovered, and evolved safely? | Data model, lineage, retention policy, backup and restore evidence, reconciliation reports, schema migration plan. |
| Operability and Observability | Can the team detect, diagnose, mitigate, and learn from production behavior? | Dashboards, alerts, logs, traces, SLOs, runbooks, incident tickets, support model. |
| Testing and QA Standards | Is release quality supported by the right automated and manual evidence? | Test strategy, automated results, non-functional test evidence, quality gates, defect trends. |
| Compliance with Enterprise Standards | Is the application conformant with required enterprise, regulatory, and platform controls? | Standards mapping, exception register, audit evidence, policy sign-offs, service acceptance checklist. |
| Maintainability and Evolvability | Can the application be changed safely and economically over time? | Code quality metrics, dependency age, modularity evidence, documentation, ownership model, technical debt register. |
| Cost, Sustainability, and Resource Efficiency | Is the workload economically and environmentally efficient without undermining resilience or performance? | Cost reports, rightsizing analysis, scaling data, lifecycle policies, utilization metrics. |

## Architecture Fitness and Standard Patterns

Review whether the architecture is coherent, intentionally designed, and aligned to enterprise architecture principles. The design should show clear separation of concerns, explicit ownership boundaries, dependency direction, platform fit, and rationale for deviations from standard patterns. ISO/IEC 25010 frames software quality in terms of product characteristics and quality-in-use characteristics, making it a useful baseline for checking whether stated requirements cover non-functional quality comprehensively ([ISO](https://www.iso.org/standard/35733.html)).

Reviewers should inspect whether design decisions are captured in architecture decision records and whether those decisions still match the implemented codebase and deployed infrastructure. Standard patterns should not be treated as dogma; deviations are acceptable when justified by business context, operational constraints, risk appetite, and evidence.

Typical review questions:

- Are bounded contexts, service ownership, data ownership, and interface boundaries explicit?
- Are chosen patterns consistent with enterprise reference architectures and platform guardrails?
- Are there hidden point-to-point dependencies, shared databases, cyclic dependencies, or “temporary” tactical integrations that have become strategic?
- Are architectural decisions documented, current, and traceable to business and operational requirements?
- Does the codebase structure reflect the architecture, or does implementation evidence contradict the design?

Common risks:

- Architecture diagrams omit runtime dependencies, batch jobs, manual operations, or third-party services.
- Components are coupled through shared database tables, undocumented message schemas, or implicit file drops.
- The selected technology stack is unsupported, has unclear ownership, or requires skills not available in the support model.
- Approved patterns are followed syntactically but not semantically, for example using queues without idempotency, monitoring, replay, or dead-letter handling.

Further reading:

- [ISO/IEC 25010 software quality models](https://www.iso.org/standard/35733.html)
- [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/)
- [AWS Well-Architected Reliability Pillar](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/welcome.html)

## Integration and API Design

Review whether integration choices are explicit and appropriate for the user journey, consistency model, latency tolerance, failure modes, and operational ownership. Synchronous request-reply can be appropriate where the caller needs an immediate result and the dependency is within the latency and availability budget, while asynchronous communication is usually better for long-running work, load leveling, decoupling, or workflows where immediate completion is not required. AWS describes asynchronous communication as a style where the client issues a request but does not receive an immediate response, with patterns such as fire-and-forget, claim check, callback, and bidirectional communication ([AWS Prescriptive Guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-integrating-microservices/asynchronous.html)).

Blocking behavior should be reviewed at the thread, connection, and user-experience levels. A design that is “async” at the messaging layer can still create blocking behavior if callers synchronously wait, queues are drained by a single worker, or consumers perform long-running operations without concurrency control.

Typical review questions:

- Why is each interaction synchronous, asynchronous, event-driven, batch, or streaming?
- What are the timeout values, retry policies, backoff rules, and cancellation semantics for each remote call?
- Are APIs versioned, documented, and protected by backward-compatible contract rules?
- Are event schemas stable, discoverable, versioned, and compatible with consumer evolution?
- Are duplicate messages, out-of-order messages, poison messages, partial failures, and replay scenarios handled?
- Are idempotency keys, correlation IDs, causation IDs, and trace context propagated?
- Is there an explicit consistency model, such as strong consistency, eventual consistency, read-your-writes, or compensating transactions?

Common risks:

- Long-running operations are implemented as synchronous HTTP calls, increasing timeout and user-experience risk.
- Retry behavior is applied at several layers, producing retry storms and long tail latency.
- Events are published before the state change is durably committed, creating lost or inconsistent notifications.
- Consumers rely on undocumented fields or ordering guarantees that the producer does not provide.
- Error handling differs by integration path, making support outcomes unpredictable.

Further reading:

- [Azure Asynchronous Request-Reply pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/asynchronous-request-reply)
- [AWS asynchronous communication guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-integrating-microservices/asynchronous.html)
- [CloudEvents specification overview](https://cloudevents.io)

## Resilience and Dependency Failure

Review how the application behaves when dependencies are slow, unavailable, throttling, returning invalid data, or failing intermittently. Resilience is not just uptime; it includes graceful degradation, fault isolation, data recovery, operational recovery, and user-safe failure behavior. Microsoft’s Azure architecture guidance describes the Retry pattern as a way to handle anticipated temporary failures when connecting to a service or network resource, and it warns that retries should be implemented only where the full context of the failing operation is understood ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/patterns/retry)).

Circuit breakers should be considered for dependencies where persistent failures or slow responses would otherwise consume resources and cause cascading failure. Microsoft describes the Circuit Breaker pattern as temporarily blocking access to a faulty service after detecting failures, preventing repeated unsuccessful attempts and allowing recovery ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/patterns/circuit-breaker)).

Typical review questions:

- What are the critical dependencies, and what are their failure modes?
- Are timeout, retry, circuit breaker, rate limiting, bulkhead, fallback, and load-shedding policies defined and tested?
- Is retry behavior bounded, jittered, observable, and safe for non-idempotent operations?
- Can the service degrade gracefully without corrupting data or misleading users?
- Are dead-letter queues monitored, triaged, replayable, and protected against infinite poison-message loops?
- Are backup, restore, failover, rollback, and disaster recovery procedures tested under realistic constraints?
- Are recovery time objectives and recovery point objectives documented and achieved in evidence?

Common risks:

- No explicit dependency timeout, causing threads or connections to block until platform defaults expire.
- Unbounded retries amplify outages or trigger rate limits.
- Circuit breakers are absent for high-latency or fragile dependencies.
- Fallbacks return stale or default data without user-visible caveats or auditability.
- DR plans exist as documents but have not been tested end-to-end.

Further reading:

- [Azure Retry pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/retry)
- [Azure Circuit Breaker pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/circuit-breaker)
- [Azure reliability design patterns](https://learn.microsoft.com/en-us/azure/well-architected/reliability/design-patterns)
- [Google SRE practices](https://sre.google/sre-book/part-III-practices/)

## Performance and Latency

Review whether performance requirements are explicit, measurable, and tested under realistic conditions. Performance should be assessed using response-time percentiles, throughput, concurrency, resource consumption, queueing behavior, cache effectiveness, downstream latency, and tail-latency behavior, not just averages. Google’s SRE guidance notes that user-facing serving systems generally care about availability, latency, and throughput, and that high variance in response times can materially affect user experience ([Google SRE](https://sre.google/sre-book/service-level-objectives/)).

Performance design should include budgets for each tier of the request path. A service cannot meet a 300 ms end-to-end objective if upstream gateways, authentication, network hops, database calls, and downstream services cumulatively consume the budget.

Typical review questions:

- What are the latency objectives for critical user journeys and API operations?
- Are objectives specified by percentile, workload class, payload size, geography, and measurement point?
- What is the measured throughput under normal, peak, and stress conditions?
- Where are the bottlenecks: CPU, memory, I/O, database locks, thread pools, connection pools, network, downstream dependency, or queue lag?
- Are expensive operations cached, indexed, batched, paginated, streamed, or moved out of the critical path?
- Is performance monitored continuously in production and compared to test evidence?

Common risks:

- Performance tests use unrealistic data volumes, too few concurrent users, or non-production infrastructure that invalidates conclusions.
- Only average response time is reported, hiding tail latency and outliers.
- Database queries lack indexes, use unbounded scans, or couple unrelated user journeys through shared locks.
- Synchronous dependency chains make latency additive and fragile.
- Test results are not linked to release candidates or production telemetry.

Further reading:

- [Google SRE Service Level Objectives](https://sre.google/sre-book/service-level-objectives/)
- [Azure Well-Architected Framework performance efficiency pillar](https://learn.microsoft.com/en-us/azure/well-architected/)
- [AWS Well-Architected Framework Reliability Pillar](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/welcome.html)

## Capacity and Scalability

Review whether the application can handle expected demand, growth, spikes, and saturation. Capacity should be modeled from business volumes, traffic patterns, batch windows, data growth, retention policies, dependency quotas, and operational limits. Scalability should include both application scaling and bottlenecks in dependencies such as databases, third-party APIs, queues, network egress, identity providers, and observability pipelines.

Capacity evidence should distinguish between normal load, peak load, stress load, and failure-mode load. Failure-mode capacity matters because retries, failover, queue buildup, and degraded dependencies can produce traffic shapes that normal tests do not simulate.

Typical review questions:

- What is the workload model, and how was it derived from business assumptions?
- What are the known platform quotas, dependency limits, connection pool limits, thread pool limits, queue throughput limits, and rate limits?
- Can scaling happen horizontally, vertically, elastically, or only through manual intervention?
- Are autoscaling thresholds based on user impact and saturation indicators rather than CPU alone?
- How long can queues absorb load before breaching service objectives?
- What happens after a downstream outage when backlogged work resumes?

Common risks:

- Capacity planning ignores batch, reporting, reconciliation, analytics, and operational traffic.
- Autoscaling is configured but not tested with realistic startup time, warmup, or dependency limits.
- Queue-based designs lack queue-age, processing-lag, and dead-letter monitoring.
- Scaling one component overloads a shared dependency.
- Cloud quotas or third-party rate limits are discovered only during production incidents.

Further reading:

- [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/)
- [Google SRE Monitoring Distributed Systems](https://sre.google/sre-book/monitoring-distributed-systems/)
- [AWS Well-Architected Reliability Pillar](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/welcome.html)

## Security and Privacy

Review whether security and privacy are designed into the system lifecycle, not bolted on after delivery. NIST SP 800-160 addresses the engineering-driven perspective and actions necessary to develop more defensible and survivable systems ([NIST CSRC](https://csrc.nist.gov/pubs/sp/800/160/v1/upd2/final)). NIST SP 800-218 recommends the Secure Software Development Framework as a core set of high-level secure software development practices that can be integrated into SDLC implementations ([NIST CSRC](https://csrc.nist.gov/pubs/sp/800/218/final)).

Application security verification should be evidence-based. OWASP ASVS provides a basis for testing web application technical security controls and for giving developers secure development requirements ([OWASP](https://owasp.org/www-project-application-security-verification-standard/)).

Typical review questions:

- Is there a current threat model covering trust boundaries, abuse cases, privileged operations, and data flows?
- Are authentication, authorization, session management, service-to-service identity, and least privilege implemented consistently?
- Are secrets stored, rotated, accessed, and audited through approved mechanisms?
- Is data encrypted in transit and at rest, with key ownership and rotation defined?
- Are vulnerabilities found through SAST, DAST, SCA, container scanning, IaC scanning, penetration testing, or manual review?
- Are privacy requirements such as minimization, consent, retention, deletion, masking, and purpose limitation implemented?
- Are logging and monitoring designed to avoid leaking sensitive data?

Common risks:

- Authorization checks differ between UI, API, batch, and administrative paths.
- Service accounts are overprivileged or shared across environments.
- Sensitive data appears in logs, traces, analytics exports, or dead-letter queues.
- Dependency vulnerabilities are accepted without expiry, compensating control, or upgrade plan.
- Threat models are outdated or disconnected from actual deployment topology.

Further reading:

- [NIST SP 800-160 Systems Security Engineering](https://csrc.nist.gov/pubs/sp/800/160/v1/upd2/final)
- [NIST SP 800-218 Secure Software Development Framework](https://csrc.nist.gov/pubs/sp/800/218/final)
- [OWASP Application Security Verification Standard](https://owasp.org/www-project-application-security-verification-standard/)

## Data Management and Integrity

Review whether data is correctly owned, protected, transformed, reconciled, retained, and recovered. The design should identify authoritative sources, replication paths, schema ownership, data quality rules, audit requirements, retention periods, deletion flows, backup strategy, and reconciliation controls.

For distributed systems, reviewers should inspect consistency assumptions explicitly. Event-driven and asynchronous designs often introduce eventual consistency, duplicate processing, delayed propagation, and reordering; these are acceptable only when the user journey, support model, reconciliation controls, and reporting semantics can tolerate them.

Typical review questions:

- What system is the authoritative source for each critical data entity?
- What consistency model applies to each read and write path?
- How are schema changes versioned, migrated, and rolled back?
- How are duplicates, missing events, late events, and out-of-order updates detected and corrected?
- Are audit trails tamper-evident, complete, and retained for the required period?
- Are backup and restore tests performed with production-like data volumes and access controls?
- Are data retention, deletion, archival, masking, and legal hold requirements implemented?

Common risks:

- Multiple services can update the same data without clear ownership.
- Event payloads lack schema governance or compatibility checks.
- Reports use eventually consistent replicas without disclosure or reconciliation.
- Backup exists, but restore time or restored data integrity is unverified.
- Sensitive production data is copied into non-production environments without masking.

Further reading:

- [ISO/IEC 25010 software quality models](https://www.iso.org/standard/35733.html)
- [AWS asynchronous communication guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-integrating-microservices/asynchronous.html)
- [CloudEvents specification overview](https://cloudevents.io)

## Operability and Observability

Review whether the application is supportable in production. Operability includes deployment, rollback, monitoring, alerting, diagnosis, incident response, support ownership, runbooks, change management, and continuous improvement. Google’s SRE guidance identifies latency, traffic, errors, and saturation as the four golden signals for monitoring user-facing systems ([Google SRE](https://sre.google/sre-book/monitoring-distributed-systems/)). OpenTelemetry defines observability APIs, SDKs, and tools for collecting telemetry such as traces, metrics, and logs ([OpenTelemetry](https://opentelemetry.io/docs/)).

Observability should be designed around user journeys and service objectives, not merely infrastructure health. Dashboards should answer whether users are affected, which component is responsible, what changed, and what action should be taken.

Typical review questions:

- Are logs, metrics, traces, and business events correlated with request IDs, trace IDs, tenant IDs, and user journey context where appropriate?
- Are service-level indicators and objectives defined for critical journeys?
- Do alerts map to actionable runbooks and known operational owners?
- Are dashboards meaningful during incidents, or do they require expert tribal knowledge?
- Are deployment, rollback, scaling, failover, and certificate or secret rotation procedures documented and tested?
- Are incidents reviewed, root causes tracked, and corrective actions completed?

Common risks:

- Monitoring detects host health but not user-impacting failures.
- Alerts are noisy, non-actionable, missing runbooks, or routed to unclear owners.
- Logs are too sparse for diagnosis or too verbose to search effectively.
- Distributed traces are absent across service boundaries, making dependency latency invisible.
- Runbooks assume manual steps that only one engineer knows how to perform.

Further reading:

- [Google SRE Monitoring Distributed Systems](https://sre.google/sre-book/monitoring-distributed-systems/)
- [Google SRE Service Level Objectives](https://sre.google/sre-book/service-level-objectives/)
- [OpenTelemetry documentation](https://opentelemetry.io/docs/)

## Testing and QA Standards

Review whether the team has adequate evidence that the application meets functional and non-functional quality expectations. Non-functional testing should cover performance, load, stress, reliability, security, usability, maintainability, compatibility, resilience, recovery, and operability where applicable. ISTQB-oriented testing guidance describes non-functional testing as testing “how” the system works and notes that it includes areas such as performance, load, stress, usability, maintainability, reliability, and portability ([Zef ISTQB Foundation](http://zef-istqb-foundation.weebly.com/23-test-types.html)).

Testing evidence should be tied to the release candidate. A strong test strategy includes automated verification in CI/CD, representative environments, production-like data where lawful and safe, service virtualization where dependencies cannot be exercised, and post-release validation.

Typical review questions:

- Does the test strategy map tests to functional requirements, non-functional requirements, risks, and acceptance criteria?
- Are unit, integration, contract, end-to-end, performance, security, accessibility, resilience, and regression tests appropriately balanced?
- Are tests deterministic, maintainable, and integrated into quality gates?
- Are non-functional tests run in production-like conditions, with realistic data, traffic, dependency behavior, and infrastructure?
- Are known test gaps explicitly risk-assessed?
- Are defects, escaped defects, flaky tests, and test coverage trends tracked?

Common risks:

- Non-functional testing is performed once, late, and outside the release pipeline.
- Contract tests are missing for service and event integrations.
- Performance testing excludes authentication, authorization, logging, encryption, and downstream dependencies.
- Test environments are too small or too different from production to support conclusions.
- QA sign-off is based on activity completion rather than evidence against acceptance criteria.

Further reading:

- [ISO/IEC 25010 software quality models](https://www.iso.org/standard/35733.html)
- [OWASP ASVS](https://owasp.org/www-project-application-security-verification-standard/)
- [NIST SSDF](https://csrc.nist.gov/pubs/sp/800/218/final)

## Compliance with Enterprise Standards

Review whether the application complies with enterprise policies, platform standards, architecture principles, security controls, operational controls, and regulatory obligations. Compliance should be demonstrated through evidence rather than asserted through self-attestation alone. ITIL 4 service design guidance states that service design should ensure services are fit for purpose and fit for use, incorporate standards and non-functional requirements, and govern the holistic design approach ([AXELOS](https://www.axelos.com/resource-hub/practice/service-design-itil-4-practice-guide)).

Standards exceptions should be explicit, time-bound, risk-accepted, and supported by compensating controls. Reviewers should distinguish between a justified exception and an undocumented gap.

Typical review questions:

- Which enterprise standards apply based on service tier, data classification, technology stack, geography, and regulatory exposure?
- Is there evidence of compliance with cloud landing-zone, network, identity, logging, encryption, data retention, and secure SDLC controls?
- Are exceptions documented with owner, reason, risk, compensating control, expiry, and remediation path?
- Are audit logs complete, protected, searchable, and retained?
- Are suppliers and third-party services assessed and governed?

Common risks:

- Application teams treat standards as advisory when they are mandatory controls.
- Exceptions have no expiry or remediation plan.
- Evidence exists in multiple tools but is not traceable to the application or release.
- Production configuration drifts away from approved baseline.
- Supplier controls are assumed rather than verified contractually or technically.

Further reading:

- [AXELOS ITIL 4 Service Design practice guide](https://www.axelos.com/resource-hub/practice/service-design-itil-4-practice-guide)
- [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/)
- [NIST Secure Software Development Framework](https://csrc.nist.gov/pubs/sp/800/218/final)

## Maintainability and Evolvability

Review whether the codebase, architecture, and team model allow safe change over time. Maintainability includes modularity, analyzability, testability, dependency hygiene, documentation, upgrade path, code ownership, and technical debt control. ISO/IEC 25010 includes maintainability and portability among product quality characteristics, supporting their inclusion as formal non-functional quality concerns ([ISO](https://www.iso.org/standard/35733.html)).

Reviewers should look beyond static code metrics. A maintainable system has clear ownership, good tests, understandable architecture, controlled dependencies, repeatable environments, and known seams for change.

Typical review questions:

- Is the codebase modular enough to change one capability without unintended impact elsewhere?
- Are dependencies current, supported, licensed, and vulnerability-managed?
- Are build, test, deployment, and local development workflows documented and reproducible?
- Are architecture diagrams and ADRs updated when implementation changes?
- Is technical debt visible, prioritized, and tied to risk or delivery impact?
- Are skills and ownership distributed enough to avoid key-person dependency?

Common risks:

- Shared libraries or frameworks create hidden coupling across many teams.
- Large modules have unclear boundaries and weak automated tests.
- Dependencies are pinned to unsupported versions or unmaintained projects.
- Documentation describes the intended design rather than actual behavior.
- The team depends on one specialist for releases, incidents, or critical design knowledge.

Further reading:

- [ISO/IEC 25010 software quality models](https://www.iso.org/standard/35733.html)
- [NIST SSDF](https://csrc.nist.gov/pubs/sp/800/218/final)

## Cost, Sustainability, and Resource Efficiency

Review whether the workload uses resources efficiently while meeting resilience, performance, and compliance requirements. Cost optimization should not mean reducing necessary resilience; it should mean matching resources to measured demand, controlling waste, understanding unit economics, and using lifecycle policies. Microsoft’s Azure Well-Architected Framework includes cost optimization, operational excellence, performance efficiency, reliability, and security as core pillars ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/well-architected/)).

Resource efficiency also supports sustainability by reducing idle compute, unnecessary storage retention, inefficient data movement, and overprovisioned environments. Reviewers should consider whether non-production environments, logs, metrics, traces, backups, and retained datasets have appropriate lifecycle policies.

Typical review questions:

- Are cost drivers understood by component, tenant, environment, transaction, and data volume?
- Are resources right-sized based on measured utilization and service objectives?
- Are autoscaling policies cost-aware while preserving performance and resilience?
- Are storage, logs, backups, and analytics datasets retained only as long as required?
- Are expensive dependencies called efficiently, cached appropriately, or isolated from high-volume paths?
- Are cost anomalies monitored and owned?

Common risks:

- Overprovisioning hides performance problems and creates avoidable run cost.
- Underprovisioning saves money but breaches latency or resilience objectives.
- Observability data grows without retention limits or sampling strategy.
- Batch and analytics workloads compete with user-facing workloads.
- Cost allocation is too coarse to drive engineering action.

Further reading:

- [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/)
- [AWS Well-Architected Framework Reliability Pillar](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/welcome.html)

## Suggested Category Scoring Criteria

Use these criteria to keep RAG ratings consistent across applications:

| Rating | Criteria |
| --- | --- |
| Green | Requirements are explicit; design is aligned to standards; implementation matches design; evidence is current; production telemetry supports claims; residual risk is acceptable. |
| Amber | Some evidence is incomplete, design is partially aligned, or implementation has manageable gaps; workaround, owner, due date, and risk acceptance are documented. |
| Red | Material gap threatens release, service objectives, security, compliance, data integrity, recoverability, or supportability; evidence is absent or contradicts claims; remediation is required. |
| Grey | Category is not applicable, not reviewed, or cannot be rated due to missing evidence. |

## Recommended Review Workflow

1. Confirm service tier, business criticality, data classification, regulatory exposure, and release milestone.
2. Validate the evidence pack before the workshop and mark missing artifacts as Grey pending evidence.
3. Review each category against design, implementation, test evidence, and production telemetry where available.
4. Capture findings in the matrix using consistent category names, risk themes, RAG values, and pattern tags.
5. Hold the governance decision separately from evidence collection so that business risk acceptance is explicit.
6. Track Red and Amber findings to closure, risk acceptance, or re-rating with new evidence.
7. Run portfolio read-across monthly or quarterly to identify systemic pattern weaknesses and update enterprise standards.
